# Stub Generator Design

**Date:** 2026-05-08
**Status:** Superseded

> **Note:** This document reflects the original approved design. The implementation diverged significantly during development. Key differences from the current codebase:
> - `SQLTable` was removed; schema reflection now uses SQLAlchemy's `AutomapBase`
> - `StubGenerator` no longer accepts `output_path`; the stub is always written next to the server's source file via `inspect.getfile`
> - Table stub classes no longer inherit from `SQLTable`
> - Generated stub imports `SQLDatabase` only (not `SQLTable`)
> - `server.all_databases_list` replaced by `server.get_available_dbs()`
> - Schema introspection uses `db._base.metadata.tables` instead of `get_column_list()` per table
> - `Server` is no longer abstract; dialects are registered in `DRIVER_REGISTRY` / `DB_SELECT_REGISTRY`

## Overview

Add a `StubGenerator` utility to quickdb that connects to a live server instance, introspects its schema, and writes a `.pyi` stub file next to the child server's `server.py`. The stubs give IDEs full autocomplete for the `server.db.table` access hierarchy and typed `fields` parameters on query methods.

## Context

quickdb uses `setattr` at runtime to attach databases, tables, and columns dynamically. This gives flexibility but no IDE visibility. The suitecrm-py `generate_stubs.py` solves the same problem for a REST API client by connecting to a live instance, discovering its schema, and writing `.pyi` files. This design applies that pattern to quickdb's SQL hierarchy.

## Architecture

The generator lives in quickdb as a reusable module. Child projects supply a thin wrapper script that instantiates their server and calls it.

```
quickdb/
  src/quickdb/scripts/
    stub_generator.py       ← new: StubGenerator class

child-server-project/
  scripts/
    generate_stubs.py       ← thin wrapper: instantiate server, call StubGenerator
  src/myserver/
    server.py
    server.pyi              ← generated output
```

**Runtime flow:**
1. Child project's `scripts/generate_stubs.py` loads credentials from `.env`, instantiates the server, passes it to `StubGenerator` with an output path.
2. `StubGenerator.generate()` walks `server.my_databases` for deep introspection and uses `server.all_databases_list` to identify surface-tier databases.
3. For each deep database: calls `load_table_list()`, then `get_column_list()` per table.
4. Writes one `.pyi` file to the specified output path.

## Two-Tier Stub Strategy

### Deep tier — `server.my_databases`

Full schema introspection down to column level. Generates:
- A `_<PascalCase>Table(SQLTable)` class per table with column attributes and a typed `limited_query` overload
- A `_<PascalCase>Db(SQLDatabase)` class per database with typed table attributes
- The server attribute typed as the specific generated database class

### Surface tier — `all_databases_list` minus `my_databases`

No introspection. Generates only:
- The database as an attribute on the server typed as `SQLDatabase`
- An inline comment: `# available — add to my_databases for full stubs`

Surface attributes make all server databases discoverable without the cost of full introspection. They do not exist at runtime (no `load_database` was called); users access non-initialized databases via `server.query_to_pandas(sql_file_path)` instead.

## Naming Convention

Class names are derived by PascalCasing the raw database or table name, then appending `Db` or `Table`:

| Raw name       | Generated class      |
|----------------|----------------------|
| `cases`        | `_CasesTable`        |
| `case_updates` | `_CaseUpdatesTable`  |
| `suitecrm`     | `_SuitecrmDb`        |
| `my_warehouse` | `_MyWarehouseDb`     |

No special-casing or suffix stripping. The `_` prefix signals auto-generated; users should not subclass or import these classes directly.

## Generated Stub Structure

```python
# AUTO-GENERATED by quickdb StubGenerator — do not edit by hand
# Generated: 2026-05-08T12:00:00

import pandas as pd
from typing import Literal
from quickdb import SQLDatabase, SQLTable, MariaDBServer

# ── Deep stubs (my_databases) ────────────────────────────────────

class _CasesTable(SQLTable):
    id: str
    name: str
    entered_date: str

    def limited_query(
        self,
        fields: list[Literal['id', 'name', 'entered_date']] | None = None,
        where: list[str] | None = None,
        limit: int = 10,
    ) -> pd.DataFrame: ...

class _SuitecrmDb(SQLDatabase):
    cases: _CasesTable
    accounts: _AccountsTable

# ── Server ───────────────────────────────────────────────────────

class SuiteCRMServer(MariaDBServer):
    # fully stubbed
    suitecrm: _SuitecrmDb

    # available — add to my_databases for full stubs
    information_schema: SQLDatabase
    performance_schema: SQLDatabase
    mysql: SQLDatabase
```

Column attributes are typed as `str`. Database-native column types are not surfaced at this stage.

## Child Project Script Interface

```python
# scripts/generate_stubs.py
from dotenv import load_dotenv
load_dotenv()

from myserver.server import SuiteCRMServer
from quickdb.scripts.stub_generator import StubGenerator
import os

with SuiteCRMServer(
    server_name=os.getenv('DB_HOST'),
    username=os.getenv('DB_USERNAME'),
    password=os.getenv('DB_PASSWORD'),
) as server:
    StubGenerator(server, output_path='src/myserver').generate()
```

## StubGenerator Class Interface

```python
class StubGenerator:
    def __init__(self, server: Server, output_path: str | Path) -> None: ...
    def generate(self) -> Path: ...  # returns path to written .pyi file
```

`generate()` returns the path of the written file so the caller can log it.

**Output filename:** derived from the server class name converted to snake_case, e.g. `SuiteCRMServer` → `suite_crm_server.pyi`. This ensures the `.pyi` sits alongside a conventionally-named `suite_crm_server.py`.

**Parent class import:** the generator inspects `type(server).__bases__[0]` to determine the actual parent class (e.g. `MariaDBServer`, `SQLServer`) and emits the correct import in the stub header rather than a hardcoded value.

## What Is Out of Scope

- Column type mapping (all columns typed as `str`)
- `top()` / `order_by` enhancements to `SQLTable` — addressed in a follow-on design
- Cookiecutter scaffolding for child projects — decision: not needed
- Stubs for databases not in `all_databases_list`
